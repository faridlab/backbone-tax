# Contributing — backbone-tax

> **Reader:** Contributor. **Mode:** how-to. Set up, make a change the regenerator won't eat, prove
> it, and open a PR that lands on the first review. Commands are the `metaphor` surface from the
> workspace CLAUDE.md and `metaphor --help`; where a raw `cargo` equivalent exists, prefer the
> `metaphor` one — it applies workspace policy.

## Dev setup

```bash
# from the workspace root, confirm where you are and that tooling is healthy
metaphor info            # → current project: backbone-tax (Module)
metaphor doctor          # tooling + plugin health

# inside backbone-tax/
metaphor migration run   # apply migrations to your local Postgres (needs btree_gist — see below)
metaphor dev test        # build + run the suite (golden oracle + probes)
metaphor lint check
```

You need a local PostgreSQL and the `btree_gist` extension (ADR-002's overlap constraint). The
`database.url` lives in `config/application.yml`. **Never** run `cargo build`/`cargo test` from the
*workspace* root — each project has its own `Cargo.toml`; use `metaphor build`/`metaphor test` (or
plain `cargo` from inside `backbone-tax/`).

## The one rule that governs every change: schema is the source of truth

Most of `src/` is **generated** from `schema/models/*.model.yaml`. `metaphor schema schema generate --force`
overwrites it. So:

- **Entity/field change** → edit the `*.model.yaml` first, regenerate, then write a migration. Never
  hand-edit a generated file outside markers — your edit is silently lost on the next regen.
- **Per-entity custom code** → put it inside `// <<< CUSTOM … // END CUSTOM` markers, or a sibling
  `*_custom.rs` file. Markers are preserved through regeneration.
- **The engine, write service, guarded routes, golden tests, the overlap migration, and `docs/**`**
  are declared `user_owned` in [`metaphor.codegen.yaml`](../metaphor.codegen.yaml) — regen skips them
  wholesale. That's where hand-authored behavior lives.

If you're unsure whether a file is generated: is it listed in `metaphor.codegen.yaml`? Does it say
"Generated by metaphor-schema / backbone-schema" at the top? If either, treat it as generated.

## Prove it first (the workflow this module holds itself to)

The engine is money code; it's pinned by a **numeric oracle**. Any behavior change must move the
oracle, not just the implementation:

1. Add or change the expected result in
   [`docs/business-flows/golden-cases.md`](business-flows/golden-cases.md) (TGC-*) — state the exact
   number.
2. Encode it in [`tests/tax_golden_cases.rs`](../tests/tax_golden_cases.rs) (engine math) or
   [`tests/integrity_probes.rs`](../tests/integrity_probes.rs) (route/guard behavior).
3. For business-level behavior, add a scenario to [`tests/features/tax.feature`](../tests/features/tax.feature)
   and the flow doc [`business-flows/tax.md`](business-flows/tax.md).
4. `metaphor dev test` — red, then green.

A bug fix starts with a failing test that reproduces it. A new rule starts with its golden case.

## What needs an ADR

This module records load-bearing decisions as ADRs (`docs/adr/ADR-NNN-*.md`). Write one when you
change the **boundary or an invariant**, for example: what tax may depend on, whether it posts to
the GL, how effective-dating resolves, or the money-rounding contract. ADRs are **immutable once
accepted** — supersede with a new ADR, don't edit an accepted one. Small, local changes don't need
an ADR; a new invariant does. See [ADR-001](adr/ADR-001-tax-engine-boundary.md) /
[ADR-002](adr/ADR-002-effective-window-overlap-and-inclusive-reconciliation.md) for the shape.

## Commits & branches

- **Conventional commits**: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:` — one focused
  change per commit, grouped by functionality; large files on their own commit.
- **One-line messages that state *why***, not "update" / "wip" / "fix stuff".
- **No signatures.** Never append `Co-Authored-By`, "Generated with", or any trailer.
- Branch off `main`; don't commit straight to `main`.

## PR checklist

Before you open the PR, confirm:

- [ ] Schema YAML edited first for any entity/field change; generated code regenerated (not hand-edited).
- [ ] Custom logic is inside `// <<< CUSTOM` markers or a `user_owned` file — survives regen.
- [ ] A golden case / probe / feature scenario covers the change; `metaphor dev test` is green.
- [ ] New migration is reversible (`.up.sql` **and** `.down.sql`) and qualifies tables as `tax.<table>`.
- [ ] Rate/rule changes are **effective-dated rows**, not edits to history; no overlapping windows.
- [ ] Tax still returns lines and never posts to the GL; no new Cargo edge to accounting (logical FKs only).
- [ ] `metaphor lint check` clean.
- [ ] An ADR added if a boundary or invariant changed; existing ADRs superseded, not edited.
- [ ] Conventional-commit messages, no signatures.

## Review expectations

Reviewers check the boundary first (does tax still import nothing it shouldn't, still return lines),
then the oracle (is the new number right and pinned), then regen-safety (will this survive
`--force`). A change that's correct but hand-edits generated code will be sent back — move it inside
markers or a `user_owned` file. For larger changes, the `code-review-and-quality` and `council`
skills give adversarial review of the boundary and the domain design.
